# Canon Rule #174 — Tokens Are Compile-Time Contracts

**Status:** ENFORCED  
**Severity:** BLOCKING  
**Scope:** build, tokens, architecture
**Version:** 1.0.0
**Date:** 2026-01-29

## Rule

Token values and definitions are compile-time contracts enforced by the build system.

## Requirements

### 1. Single Source of Truth

- **ALL tokens MUST be declared in `src/tokens/inventory.rs`**
- No token can exist outside this inventory
- Build MUST fail if undeclared token is used

### 2. Value Primitive Isolation

- **Literal values (numbers, colors, measurements) can ONLY exist in `primitives.css`**
- No literal in:
  - Family CSS (generated or manual)
  - Component CSS
  - Block CSS
  - Theme overrides (except semantic mapping)

### 3. Layer Contracts

**PRIMITIVES** (`primitives.css`)

- ✅ Define immutable base values
- ❌ NEVER reference UI concerns
- ❌ NEVER vary by theme

**SEMANTIC** (`semantic.css`)

- ✅ Map primitives to theme-specific meanings
- ✅ ONE definition per token per theme
- ❌ NEVER create new values

**FAMILIES** (generated by `build.rs`)

- ✅ Derive from semantic + primitives only
- ❌ NEVER define literals
- ❌ NEVER create base values

**COMPONENTS/BLOCKS** (consumption layer)

- ✅ Consume family tokens only
- ❌ NEVER define tokens
- ❌ NEVER use literals

## Enforcement

### Build-Time

```bash
cargo build --lib  # Generates whitelist from inventory.rs
npm run build      # Validates all CSS against whitelist
```

### CI Requirement

```yaml
- name: Validate Tokens
  run: |
    cargo build --lib
    npm run tokens:validate
```

Build MUST fail if:

- Token used but not in inventory
- Literal value in generated CSS
- Token defined outside primitives.css
- Duplicate token definition

## Why This Rule Exists

**Without this rule:**

- ❌ Drift between design and code
- ❌ Inconsistent values across components
- ❌ No single source of truth
- ❌ Runtime-only validation (too late)
- ❌ Human discipline = human error

**With this rule:**

- ✅ Impossible to use undeclared tokens
- ✅ Build breaks before commit
- ✅ Self-documenting system
- ✅ Refactoring is safe
- ✅ System enforces correctness

## Examples

### ❌ VIOLATIONS

```css
/* family-f-datamedia.css - WRONG */
:root {
  --data-spacing: 1.5rem; /* ❌ Literal value */
}
```

```css
/* card.css - WRONG */
[data-card] {
  padding: 1rem; /* ❌ Literal value */
  background: var(--card-custom); /* ❌ Token not in inventory */
}
```

```rust
// Using token not in inventory - WRONG
// Build succeeds, fails at validation
```

### ✅ CORRECT

```rust
// src/tokens/inventory.rs
pub const PRIMITIVE_TOKENS: &[&str] = &[
    "primitive-space-lg",
];
```

```css
/* primitives.css */
:root {
  --primitive-space-lg: 1.5rem; /* ✅ Only place for literals */
}
```

```css
/* family-f-datamedia.css (GENERATED) */
:root {
  --data-spacing: var(--primitive-space-lg); /* ✅ Token reference */
}
```

```css
/* card.css */
[data-card] {
  padding: var(--data-spacing); /* ✅ Consumes family token */
}
```

## Migration Path

Existing code violating this rule:

1. Extract all literals to `primitives.css`
2. Add tokens to `inventory.rs`
3. Run `cargo build --lib` to regenerate
4. Replace literals with token references
5. Run `npm run tokens:validate`
